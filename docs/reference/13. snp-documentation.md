# Semantic Navigation Protocol (SNP)

## Abstract

### Historical Challenges in Structured Data Processing

For decades, software systems have relied on structured and semi-structured formats such as JSON, XML, YAML, configuration files, database records, and markup documents to store and exchange information.

While these formats successfully standardized data representation, they introduced a recurring challenge:

**the inability to uniformly navigate, query, and manipulate data regardless of its structure, storage medium, or serialization format.**

Historically, each technology stack developed its own mechanisms:

| Technology    | Navigation Mechanism       |
| ------------- | -------------------------- |
| JSON          | JSONPath, object traversal |
| XML           | XPath                      |
| YAML          | Custom parsers             |
| SQL Tables    | SQL Queries                |
| Redis         | Key traversal              |
| Git Documents | File parsing               |
| Remote APIs   | Endpoint-specific logic    |

As a result:

* Query logic became tightly coupled to storage technologies.
* Application code repeatedly implemented traversal algorithms.
* Migration between storage technologies became expensive.
* Business logic became polluted with serialization concerns.
* Developers were forced to learn multiple navigation paradigms.

The challenge becomes more pronounced as modern systems increasingly combine:

* databases
* caches
* APIs
* filesystems
* repositories
* AI knowledge sources
* structured documents

within a single application architecture.

---

### Introducing SNP

The Semantic Navigation Protocol (SNP) is a unified protocol for navigating, querying, selecting, and manipulating structured data through semantic paths.

Rather than coupling operations to JSON, XML, SQL, YAML, or any specific technology, SNP introduces a generalized navigation model based on:

* semantic paths
* semantic selectors
* semantic filters
* semantic CRUD instructions

The protocol allows identical instructions to operate on:

* JSON documents
* YAML documents
* XML documents
* database rows
* cache objects
* markdown RFC documents
* remote API payloads
* biological engine DNA models

without requiring application code to understand the underlying storage format.

---

### Why SNP Is Relevant

Modern systems increasingly operate on data distributed across multiple technologies.

Examples include:

* cloud-native systems
* AI systems
* configuration management platforms
* document processing engines
* distributed microservices
* digital twin systems
* knowledge graph systems

These systems require a common abstraction capable of:

* locating information
* filtering information
* updating information
* synchronizing information

independent of storage technology.

SNP provides this abstraction.

---

### Broad Scope Coverage

Unlike traditional technologies which are format-specific, SNP is intentionally format-agnostic.

Potential SNP implementations include:

| Adapter | Target           |
| ------- | ---------------- |
| JSNP    | JSON             |
| YSNP    | YAML             |
| XSNP    | XML              |
| RSNP    | RFC Documents    |
| CSNP    | Cache Objects    |
| DSNP    | Database Records |

This design allows a single protocol to operate across multiple domains while preserving a consistent developer experience.

---

### Relevance Within Corpdesk

Within Corpdesk, SNP serves as the semantic foundation for the Biological Engine.

The Biological Engine treats RFC documents as living DNA structures that evolve, mutate, and guide system behavior.

Because RFC data may originate from:

* MySQL
* Git repositories
* local files
* remote URLs
* caches

a technology-neutral navigation protocol becomes essential.

SNP enables Corpdesk to:

* query RFC DNA
* update RFC DNA
* synchronize RFC DNA
* version RFC DNA
* execute RFC-driven system behaviors

without coupling business logic to any particular storage technology.

# Design

## Core Principle

SNP is based on semantic path navigation.

Example:

```ts
{
    path: [
        "parsedContent",
        "rfcId"
    ]
}
```

The protocol identifies data through semantic location rather than storage implementation.

---

## Architecture

```txt
                    SNPService
                         |
            -------------------------
            |                       |
      Format Adapters         Data Sources
            |                       |
      ----------------      ------------------
      |      |      |      |   |   |   |    |
    JSNP   YSNP   XSNP   MySQL File Git Cache
```

---

## SNPService

The SNPService contains protocol semantics.

Responsibilities:

* path navigation
* selector evaluation
* filter evaluation
* CRUD execution
* semantic mutations

The SNPService does not know:

* JSON
* YAML
* XML
* MySQL
* Git

These concerns are delegated to adapters and data sources.

---

## Format Adapters

Format adapters provide serialization-specific behavior.

Examples:

### JSNPAdapter

Responsible for:

* JSON traversal
* JSON mutation
* JSON serialization

### YSNPAdapter

Responsible for:

* YAML traversal
* YAML mutation
* YAML serialization

### XSNPAdapter

Responsible for:

* XML traversal
* XML mutation
* XML serialization

---

## Data Sources

Data sources provide persistence-specific behavior.

Examples:

### MysqlSnpDataSource

Responsible for:

* MySQL querying
* MySQL persistence
* MySQL path translation

### FileSnpDataSource

Responsible for:

* filesystem retrieval
* filesystem persistence

### GitSnpDataSource

Responsible for:

* repository retrieval
* repository commits
* repository synchronization

### CacheSnpDataSource

Responsible for:

* cache retrieval
* cache persistence

# Implementation In Corpdesk

## Request Structure

SNP introduces semantic operations:

```ts
{
    snpWhere: [],
    snpCrud: []
}
```

### snpWhere

Defines semantic selection criteria.

### snpCrud

Defines semantic mutations.

---

## Example Request

```ts
{
    snpWhere: [
        {
            path: [
                "cdBioEngineDnaSrc",
                "url"
            ],
            op: "eq",
            value: "https://github.com/corpdesk/cd-prompts/blob/main/docs/reference/1.%20corpdesk-rfc-0001_architecture_and_conventions.md"
        }
    ],

    snpCrud: [
        {
            path: [
                "cdBioEngineDnaSrc",
                "url"
            ],
            action: "upsert",
            value: {...}
        }
    ]
}
```

# Example Walkthrough

## Step 1

The caller submits:

```ts
snpWhere
```

to identify the target DNA.

Example:

```ts
{
    path: [
        "cdBioEngineDnaSrc",
        "url"
    ],
    op: "eq",
    value: "https://github.com/..."
}
```

This means:

"Find all DNA records whose source URL matches the supplied URL."

---

## Step 2

BaseService delegates retrieval to a datasource.

```ts
const ds =
    SnpDataSourceFactoryService.create(
        this.repo,
        {
            type: "mysql",
            jsonColumn: "cdBioEngineDnaData"
        }
    );

const fetchedData =
    await ds.find(
        selectors
    );
```

The service does not know how MySQL performs the search.

The datasource is responsible for translating SNP selectors into database-specific queries.

---

## Step 3

SnpService performs semantic mutations.

```ts
const updatedData =
    await SnpService.performCrud(
        plData.snpCrud,
        fetchedData
    );
```

The protocol applies:

```ts
{
    action: "upsert"
}
```

to the selected semantic path.

---

## Step 4

The modified DNA is persisted.

```ts
await this.updateI(
    req,
    res,
    serviceInput
);
```

At this point:

* selection was performed by SNP selectors
* mutation was performed by SNP instructions
* persistence was handled by the datasource

Each concern remains isolated.

# Conclusion

The Semantic Navigation Protocol (SNP) establishes a unified navigation and mutation model for structured information.

By separating:

* protocol semantics
* format implementations
* persistence implementations

SNP enables Corpdesk to evolve beyond JSON-centric architectures toward a generalized semantic data ecosystem capable of supporting future Biological Engine, RFC DNA, and AI-driven system architectures.

This document is already at a level where it can serve as the seed for a formal Corpdesk RFC (e.g., `corpdesk-rfc-snp-0001`) and later be expanded with protocol specifications, sequence diagrams, adapter contracts, performance benchmarks, and patent-oriented novelty sections.
