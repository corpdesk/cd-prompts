# 1. Abstract

This document defines the **AppCraft Scan Mechanism**, a core capability within the Corpdesk automation framework responsible for:

* **Analyzing an existing application directory**
* **Transforming it into a structured, machine-readable descriptor (`CdAppDescriptor`)**
* **Generating a formal directory signature (`DirectorySignatureDescriptor`)**
* **Evaluating conformity against Corpdesk architectural conventions**
* **Returning results via `ICdResponse` under the CdWire protocol**

The scan operation establishes the **foundation for automated development**, enabling:

* Structural validation
* Automated refactoring
* Reverse engineering into Corpdesk-compliant systems
* AI-assisted development workflows

# 🔹 9.5 Stage 5 — Weight Assignment (Expanded)

## 9.5.1 Overview

Weight Assignment is the process of attributing **quantitative significance** to each `DirectoryNode` based on its **architectural role**, **structural importance**, and **compliance relevance**.

Unlike conventional scanners that perform **binary validation (pass/fail)**, the AppCraft Scan introduces a **continuous scoring model**, enabling nuanced evaluation of application structures.

---

## 9.5.2 Rationale

### 1. Limitations of Conventional Scanners

Traditional directory analyzers and linters typically:

* Validate **presence/absence** of files
* Enforce **static rules**
* Produce **boolean outputs**:

  * ✅ Valid
  * ❌ Invalid

This approach has inherent limitations:

| Limitation        | Impact                                                  |
| ----------------- | ------------------------------------------------------- |
| Binary evaluation | Cannot express partial correctness                      |
| No prioritization | All violations treated equally                          |
| Low adaptability  | Difficult to apply across diverse codebases             |
| Poor guidance     | Does not indicate *how close* a system is to compliance |

---

### 2. Need for Gradual Compliance Measurement

Real-world systems:

* Are often **partially compliant**
* Evolve incrementally
* Require **progressive alignment**, not abrupt rejection

Therefore, a **graded evaluation system** is required.

---

## 9.5.3 Core Concept

Each node is assigned a **weight (W)**:

```id="4m8ybt"
DirectoryNode.weight ∈ ℝ⁺
```

Weights represent:

* Structural importance
* Architectural criticality
* Contribution to overall compliance

---

## 9.5.4 Weight Categories

| Category         | Example                              | Weight Range | Significance           |
| ---------------- | ------------------------------------ | ------------ | ---------------------- |
| Critical Nodes   | `main.ts`, `.cd/app-descriptor.json` | 9–10         | Define system identity |
| Structural Nodes | `sys/`, `app/`, `utils/`             | 6–9          | Define architecture    |
| Functional Nodes | controllers, services, models        | 7–8          | Define behavior        |
| Auxiliary Nodes  | scripts, configs                     | 3–6          | Support system         |
| Unknown Nodes    | unrecognized files                   | 0–2          | Potential deviation    |

---

## 9.5.5 Conformity as a Continuous Function

Instead of:

```id="skj5z5"
isValid = true | false
```

AppCraft defines:

```id="v0n9ql"
CR = Σ(W_matched) / Σ(W_expected)
```

Where:

* `CR ∈ [0,1]`
* Represents **degree of compliance**

---

## 9.5.6 Threshold-Based Compliance

This enables **practical decision-making**:

| CR Range    | Interpretation                |
| ----------- | ----------------------------- |
| ≥ 0.90      | Fully compliant               |
| 0.75 – 0.89 | Acceptable (minor deviations) |
| 0.50 – 0.74 | Needs alignment               |
| < 0.50      | Non-compliant                 |

### Key Advantage

> Compliance is **not binary**, but **threshold-driven**

This allows:

* Gradual adoption of Corpdesk
* Realistic evaluation of legacy systems
* Controlled enforcement policies

---

## 9.5.7 Infection Metric (Complementary)

Weighting also enables **negative signal detection**:

```id="7r3p7f"
I = V(χ) / V(total)
```

Where:

* `χ` = set of anomalous nodes

This captures:

* Structural pollution
* Non-standard patterns
* Unregistered modules
